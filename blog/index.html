Your GitHub Repos Aren't as Safe as You Think. ForkOff Fixes That.

An open-source tool that mirrors every repo you own — branches, tags, full history — to your own machine. Automatically. Every single day.

The Story Behind This

This project started the way a lot of good tools do — with someone else's nightmare. The creator, SkyzFallin, read about a developer who had their entire GitHub account suspended without warning. No email. No heads-up. Just a login page that said the account was gone for violating the terms of service. Years of repositories, contribution history, sponsors, open-source work — all of it inaccessible overnight.

And this wasn't some edge case. If you go look at the GitHub community forums, you'll find thread after thread of developers — some of them maintainers of major projects — who woke up one morning locked out of their own code. Some were reinstated after weeks of waiting. Some never got a response at all. One developer described losing not just their personal repos, but admin access to their company's entire organization, leaving every employee unable to reach the codebase. Their whole online identity as a developer, gone in an instant.

Here's the thing that makes it sting: GitHub's Terms of Service are explicit about this. They reserve the right to suspend or terminate your access at any time, with or without cause, with or without notice. That's not buried in fine print — it's in the agreement you clicked "I agree" on when you signed up. There is no guaranteed recourse. If your account gets flagged by an automated system, you're at the mercy of a support queue that can take months to respond — if it responds at all.

That's not a criticism of GitHub specifically — it's the reality of building your career on any platform you don't control. SkyzFallin decided to stop hoping it wouldn't happen and just build something that made it not matter if it did.

What ForkOff Actually Does

ForkOff is a Bash tool that runs on any Linux machine with systemd. It connects to the GitHub API, discovers every repo tied to your account — public and private — and creates full git clone --mirror backups of each one. Not shallow copies. Full mirrors with every branch, every tag, every ref, and the complete commit history.

It runs as a systemd timer, so once you set it up, it fires every day at whatever time you choose (3 AM UTC by default). New repos you create get picked up automatically on the next run. There's no config file to update when you start a new project — it just handles it.

Each run produces a JSON audit report with timestamps, pass/fail status per repo, file sizes, and branch counts. If you need to prove to a client or an auditor that daily backups are actually happening, the receipts are there. It also runs random git fsck integrity checks on your mirrors so you're not just accumulating backups that might be silently corrupted.

The error handling is solid for a Bash project. The script runs with set -euo pipefail, so it fails fast on unexpected errors instead of silently continuing with a half-broken state. Every repo gets individual pass/fail tracking in the audit report, so if one mirror fails it doesn't take down the whole run. API failures are caught and handled. A lock file prevents overlapping runs if a backup takes longer than expected. And if anything does go wrong, you can configure a desktop error notification — a plain text file that gets dropped somewhere visible so you're not relying on remembering to check logs.

The installer is interactive — it validates your GitHub token before doing anything, then gives you a TUI-based repo picker where you can exclude forks or repos you don't care about. The whole thing stays out of your way after that. Logs auto-clean after 90 days.

Getting Started

You need a Linux box with systemd and a GitHub Personal Access Token with repo scope. The installer handles installing git, curl, and jq if they're not already present.

git clone https://github.com/SkyzFallin/ForkOff.git
cd ForkOff
sudo bash install.sh

Once it launches, the installer walks you through the entire setup interactively. Here's what to expect:

Step 1 — Dependencies. It checks for git, curl, and jq. If any are missing, it installs them via apt automatically.

Step 2 — GitHub username. It asks for your GitHub username. Straightforward.

Step 3 — Personal Access Token. You paste in a GitHub PAT with repo scope. The installer immediately tests it against the GitHub API and confirms which account it authenticated as. If the token is bad, it stops here instead of letting you build a broken config. You can create one at github.com/settings/tokens — pick "Generate new token (classic)" and make sure the repo checkbox is selected. The classic token is what you want here since ForkOff uses the repo scope, which is specific to classic tokens.

Step 4 — Repo selector. This is the nice part. The installer fetches your full repo list from the API and opens a terminal-based picker. You navigate with arrow keys, press space to toggle repos you want to exclude (forks are labeled so they're easy to spot), and hit enter to confirm. Anything you don't exclude gets backed up.

Step 5 — Backup directory. It asks where you want mirrors stored. The default is /opt/github-backups, which is fine for most setups. It creates the directory structure for you — mirrors, logs, and reports.

Step 6 — Error notifications. Optionally, you can have ForkOff drop a text file on your desktop whenever a backup run encounters errors. If you're running this on a headless server, you can skip it.

Step 7 — Schedule. Pick what time of day the backup runs, in 24-hour UTC format. Default is 03:00. It sets up a systemd timer behind the scenes.

After you confirm the summary, it deploys everything: writes the config, installs the backup script to /usr/local/bin/github-backup, stores your token securely, installs the systemd units, enables the timer, and runs an initial backup immediately so you're not waiting until 3 AM to find out if it works. The whole process takes a couple of minutes, most of which is the initial clone of your repos.

If you ever need to change something later — say you created a new token, or want to exclude a repo you didn't have before — just run the installer again. It detects the existing installation, pre-fills your previous settings, and lets you update whatever you need.

Where Your Backups Live

/opt/github-backups/
  ├── mirrors/  # bare git mirrors (RepoName.git/)
  ├── logs/    # per-run logs, 90-day retention
  └── reports/ # JSON audit reports

/etc/github-backup/
  └── github-backup.conf # config (no secrets)

Your PAT is stored in a systemd environment file with chmod 600 and root-only access. It only enters memory when the backup service runs. Rotating the token is a single command.

Restoring From a Backup

If you ever need to push a mirrored repo to a new remote — say, after migrating off GitHub entirely — it's two commands:

cd /opt/github-backups/mirrors/RepoName.git
git remote set-url origin [email protected]:user/RepoName.git
git push --mirror

Everything comes back: every branch, every tag, full history. That's the advantage of a true mirror over a shallow clone or a ZIP export from GitHub's web UI.

No Versioned Backups

One thing to be clear about: ForkOff doesn't keep multiple snapshots of your repos over time. Each run updates the mirror in place — it's a sync, not an incremental or versioned backup. If a repo gets corrupted upstream and you don't catch it before the next run, the mirror gets that corruption too. There's no "roll back to last Tuesday" built in.

If you want that kind of protection — sometimes called rotating or versioned backups — you'd need to layer something on top yourself. A cron job that tars up the mirrors directory before each run, a tool like rsnapshot or restic pointed at the backup folder, or even just rsyncing to a second drive. ForkOff's job is getting your code off GitHub and onto your machine. What happens to it after that — versioning, redundancy, offsite copies — is up to you.

A Note on Encryption

ForkOff does not encrypt your backups. Your mirrors are stored as plain git repositories on disk — the same way they'd exist if you cloned them manually. For public repos, this doesn't matter. But if you have private repos that contain sensitive data — API keys, credentials, configuration secrets — those files will be sitting in cleartext on whatever machine you're backing up to.

Encryption at rest is not part of this project. That's a deliberate scope decision, not an oversight. Securing the machine where your backups live is your responsibility. Full-disk encryption (LUKS on Linux, for example), proper file permissions, and keeping the box locked down are all on you. ForkOff gets your code off GitHub and onto hardware you control — what you do with that hardware from a security standpoint is your call.

What It Doesn't Cover

Git mirrors are comprehensive for code, but there are GitHub-specific artifacts that live outside the repository itself: issues, pull requests, discussions, Actions run history, release binaries, webhooks, and deploy keys. For those, GitHub's built-in account export (Settings → Archives → Export account data) fills the gap. ForkOff handles the part you can't recreate — your actual code, branches, and tags.

Worth noting: ForkOff is MIT licensed, written entirely in Bash, and has zero external dependencies beyond standard Linux tools. It's the kind of project that does one thing well and stays out of your way.

Who Should Use This

Anyone who has real work on GitHub. Solo developers who can't afford to lose years of side projects. Freelancers who need to prove deliverables for client SLAs. Teams that want a compliance-ready audit trail showing daily backups are actually running. Or anyone who's read one too many account-suspension horror stories and decided to stop gambling with their own code.

"Back up your repos before GitHub decides they're not your repos."

It takes about five minutes to set up. You'll forget it's even running — which is exactly the point.
