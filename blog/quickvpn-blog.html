QuickVPN: your own WireGuard VPN in one command

Ninety seconds from nothing to a hardened personal VPN. No config files, no SSH gymnastics, no commercial provider you have to take at their word.

By SkyzFallin April 2026 ~6 min read MIT licensed
~/posts/quickvpn $ cat intro.md

I built QuickVPN because every time I wanted a personal VPN, I hit the same wall: spin up a server, SSH in, install WireGuard, edit config files, fight with the firewall, generate keys, copy them around, then debug why the tunnel won't come up. It's the kind of task that takes ten minutes if you've done it a hundred times — and an entire afternoon if you haven't.

QuickVPN collapses that whole process into one command:

quickvpn setup

About 90 seconds later, you have a hardened WireGuard server running on DigitalOcean and a .conf file (plus a QR code) ready to import into the WireGuard client on any device.

What it actually does

When you run quickvpn setup, it:

  1. Creates a small Ubuntu droplet on DigitalOcean (~$6/month)
  2. Connects over SSH using a temporary ED25519 key that gets deleted from your DO account afterward
  3. Installs WireGuard, generates server keys, and configures the interface
  4. Locks down the firewall — only SSH (22/tcp) and WireGuard (51820/udp) are open
  5. Installs fail2ban, disables password login, creates a non-root admin user
  6. Enables unattended-upgrades so security patches land automatically
  7. Kicks off a full apt upgrade in the background (your VPN works during it)
  8. Generates your client config locally on your machine — your private key never touches the server
  9. Drops a .conf file and an SVG QR code in your working directory

You see a live progress bar with the current step, elapsed time, and step counter. Total time typically lands between 60 and 90 seconds.

Adding more devices

Want your VPN on your phone too? After setup, run:

quickvpn add-client --open-qr

A QR code pops up in your browser. Open the WireGuard app on your phone, scan, done. The new peer is added to the live WireGuard interface without a restart — existing clients stay connected.

You can have up to 253 clients per server. Each one gets its own keypair generated locally, so compromising one device doesn't compromise the rest.

One config, one device Don't reuse the same .conf file on two devices at once. WireGuard will peer-roam to whichever device sent a packet most recently and traffic will flip-flop between them. add-client takes ten seconds — just run it again.

The security story

This is where I spent the most time. A VPN you can set up in 90 seconds is only useful if it's actually secure.

Cryptography you can't misconfigure

WireGuard's crypto is fixed and modern — Curve25519, ChaCha20-Poly1305, BLAKE2s. There are no cipher suites to misconfigure because there are no cipher suites to choose.

Strict key separation

  • Your DigitalOcean API token lives in memory only and is never written to disk. Revoke it after setup if you don't need management commands.
  • SSH keys are ephemeral, generated per session, and deleted from DigitalOcean after setup completes.
  • Client private keys are generated on your machine. The server only ever sees public keys.

Hardened before any port is exposed

  • Password authentication disabled; root SSH restricted to key-only
  • Non-root quickvpn admin user with limited sudo
  • fail2ban rate-limits SSH (5 failures = 1-hour ban)
  • UFW firewall blocks everything except SSH and WireGuard
  • WireGuard does not log traffic by design — there's nothing to disable

Want to go further? Pass --lock during setup to disable SSH entirely after provisioning. The server becomes a black box that does exactly one thing: route encrypted packets. (Heads up: this also prevents add-client and check from working, so add all the devices you need first.)

What about DNS leaks?

Every client config ships with Cloudflare DNS (1.1.1.1, 1.0.0.1) baked in, and AllowedIPs = 0.0.0.0/0, ::/0 routes all traffic — IPv4 and IPv6 — through the tunnel. Combine that with the WireGuard app's built-in kill switch (Windows, macOS, iOS, and Android all have one) and you get full-tunnel protection that fails closed.

On Linux there's no app-level switch, but a single iptables rule achieves the same thing — full docs are in the repo.

Other useful commands

quickvpn status    # list your active QuickVPN servers
quickvpn check     # SSH in and verify everything is healthy
quickvpn destroy   # tear it down and stop billing

quickvpn check is the one I use most after the initial setup — it shows ✓/✗ for each service, the latest WireGuard handshake (so you can confirm a remote client is actually connecting), the firewall state, and whether the background upgrade has finished.

quickvpn destroy gives you an interactive picker if you don't pass a droplet ID, which matters more than it sounds — nobody wants to memorize droplet IDs at 11pm when they're trying to stop a bill.

Why I built it

Commercial VPN providers ask you to trust them with your traffic. They have access to your sessions, your DNS queries, and whatever metadata they choose to log. Several have been caught lying about their no-log policies, and a significant share of the popular "privacy" VPNs on the market are quietly owned by holding companies in jurisdictions with opaque data-handling laws — or by parent companies that have been linked to foreign intelligence services. You are, in effect, picking whose side-channel to pipe your traffic through.

Beyond ownership, there's the concentration risk. Commercial VPNs are fat targets: one breach of a shared server compromises millions of users at once. It has happened more than once. When a provider's infrastructure is popped, your traffic is part of the haul whether you were the target or not.

A personal VPN inverts all of that:

  • You own the logs. WireGuard doesn't write them, and nobody can subpoena, sell, or leak what doesn't exist. There is no dashboard, no analytics pipeline, no billing system correlating your traffic to your identity.
  • You own the server. One user, one droplet. No shared infrastructure means no blast radius from someone else's compromise. If your server ever gets popped, the damage is bounded to you — and you can quickvpn destroy and rebuild in 90 seconds.
  • You own the jurisdiction. You pick the DigitalOcean region when you run setup. No hidden parent company in a country you've never heard of, no mystery board of directors, no quiet change of ownership in a press release you'll never read.

The downside has always been the setup friction. QuickVPN is my attempt to remove that friction without cutting any corners on hardening.

One more tip Once your VPN is running, you don't need the DigitalOcean API token anymore. Revoke it. Or, if you want to keep management access, create a second read-only token for quickvpn status and only use a full-access token when you're actually provisioning.

Try it

You need Python 3.9+, a DigitalOcean account, and an API token. Then:

$ pipx install .
$ export DO_API_TOKEN="dop_v1_..."
$ quickvpn setup
